# Developer Guide

## Overview

`wfu-wp-cli` (published as `wfuwp` on npm) is a comprehensive Node.js command-line interface for managing Wake Forest University's WordPress multisite installation across multiple AWS environments (dev, uat, pprd, prod). It automates S3 file synchronization, EC2 instance management, SSH connections, DNS spoofing for development, ClickUp task management, local development environment setup via DDEV, database restore operations, and release branch cleanup workflows.

## Architecture

### Technology Stack

- **Language**: TypeScript, compiled to JavaScript
- **CLI Framework**: Commander.js
- **Testing**: Jest with ts-jest
- **Linting**: ESLint with TypeScript plugin
- **Formatting**: Prettier
- **HTTP Client**: Axios (for ClickUp API)
- **Terminal Output**: Chalk for colored output
- **User Input**: Inquirer.js for interactive prompts
- **Git Operations**: simple-git

### Command Architecture

Each command is implemented as a separate TypeScript module in `src/commands/`. The main entry point (`src/index.ts`) registers all commands with Commander.js. Commands follow a consistent pattern:

1. Define command name, arguments, and options using Commander.js API
2. Validate inputs (environment names, site IDs, required configuration)
3. Execute the operation with appropriate safety checks (confirmations, dry-run support)
4. Report results with colored terminal output

### Configuration System

Configuration is stored in `~/.wfuwp/config.json` with encrypted password fields. The system supports:

- Multi-environment database settings (host, user, password, database per environment)
- S3 bucket configuration for backups
- Local backup path configuration
- ClickUp API token storage (encrypted)
- WordPress path for release operations
- Interactive configuration wizard (`wfuwp config wizard`)

## Data Flow

### S3 Sync Flow

1. User specifies site ID, source environment, and destination environment.
2. Input validation ensures valid environments and site IDs.
3. Confirmation prompt (unless `--force` flag used).
4. AWS CLI `s3 sync` command is executed between the appropriate S3 buckets.
5. Results are displayed with colored output.

### ClickUp Integration Flow

1. API token is stored encrypted in local configuration.
2. Axios HTTP client communicates with ClickUp API v2.
3. Task data is formatted for display using chalk-colored output.
4. Export functionality supports CSV, JSON, and Markdown formats.

### Local Development Flow

1. `local install` checks and installs prerequisites (Docker, DDEV).
2. `local domain` manages `/etc/hosts` entries for local subdomains.
3. `local start/stop/restart` controls DDEV containers.
4. `local refresh` downloads fresh database from S3 production backups.

## Development Setup

```bash
git clone <repository-url>
cd wfu-wp-cli
npm install
npm run build
npm link    # Makes `wfuwp` available globally
```

### Development Scripts

```bash
npm run build      # Compile TypeScript to dist/
npm run dev        # Run with ts-node for development testing
npm run test       # Run Jest test suite
npm run lint       # Run ESLint on source files
npm run format     # Format code with Prettier
npm run release    # Execute release script
```

## Testing

Tests are located in `tests/` with corresponding `.test.ts` files for each command.

```bash
# Run all tests
npm test

# Run a specific test
npm test -- --testNamePattern="test name"
```

The project uses Jest with ts-jest for TypeScript test support.

## Coding Standards

- TypeScript with strict mode
- ESLint with TypeScript plugin and Prettier integration
- Consistent use of async/await for asynchronous operations
- Colored terminal output via Chalk for user feedback
- Interactive confirmations for destructive operations
- Dry-run support for operations that modify data

## Contributing

1. Create a feature branch from `main`.
2. Make changes following the existing patterns.
3. Add tests for new commands or utilities.
4. Run `npm test` and `npm run lint` to verify.
5. Update `CHANGELOG.md` with a description of changes.
6. Bump the version in `package.json`.
7. Open a pull request against `main`.

### Release Process

The `scripts/release.sh` script handles version bumping, tagging, and npm publishing. Run via `npm run release`.
